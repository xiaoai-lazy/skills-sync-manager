# v0.7 Convenience Operations Design

## Goal

Improve day-to-day convenience for project targets: sync an installed-skill set from another target in the same project, show per-source installed/total counts and prefer installed skills at the top of the target skill list, and expose Hub upload on both Skill Hub tabs when a Hub root is selected.

## Scope

In scope:

1. Cross-target installation sync (link-based, same project only), via a post-create follow-up and a target-page action.
2. Target detail source-tree counts (`installed/total`) and right-list sort by skill name.
3. Skill Hub «上传到 Hub» visibility on the discover (`可安装`) tab under the same selection rules as today.

Out of scope:

- Physical file copy between target directories.
- Syncing from global agents or other projects.
- Named install profiles / templates.
- Changing Skill Hub source-tree badges or hub-tab count behavior.
- Upload entry points outside Skill Hub.
- “Fully align” uninstall of extras on the destination target.

## Background

Targets store installations as junctions/symlinks into the main library (`installations[]` + `link_installer`). Creating a project target today leaves an empty installation set even when a sibling agent directory already has a useful set. Target detail shows all main-library skills with per-target install state, but the source tree has no installed/total badge and the list keeps backend `dir_name` order. Hub upload is gated to the installed tab.

## Architecture

Keep the two-layer install model. “Copy installed skills” means re-running link installs for each `skillStorageKey` present on the source target.

```
Add target success                    TargetDetail “从其他目录同步…”
        │                                        │
        └────────────┬───────────────────────────┘
                     ▼
           SyncFromTargetDialog
                     │
                     ▼
     sync_target_installations(sourceId, destId)
                     │
                     ▼
     for each source installation → install_skill(dest)
     (already installed / conflict → skipped; errors → failed[])
                     │
                     ▼
     SyncTargetInstallationsResult → UI summary → refresh target state
```

| Unit | Responsibility |
|------|----------------|
| `sync_target_installations` (Rust command) | Validate same `projectId`, source ≠ dest; batch link installs; return summary |
| `SyncFromTargetDialog` | Shared UI for post-create follow-up and target-page sync |
| `AddTargetDialog` / add-target success path | Unchanged create UX; after success, optionally open follow-up |
| `TargetDetail` | Source counts, list sort, sync entry button |
| `SkillHubPage` | Drop `tab === 'installed'` from upload visibility |

## Feature 1 — Sync Installations Across Project Targets

### Semantics

- Operation installs links on the destination; it does not copy skill files.
- Source candidates are other targets with the same `projectId` and `installedCount > 0`.
- Policy is **fill missing only**: destination skills that are already installed or in conflict are skipped. Extras on the destination are never uninstalled.
- Partial success is allowed; already-applied installs are not rolled back.

### Backend

Add a command, e.g. `sync_target_installations { source_target_id, dest_target_id }`, that:

1. Loads config and rejects if either target is missing.
2. Rejects if either target is not `scope: project`, or `projectId` differs, or ids are equal.
3. Collects distinct `skillStorageKey` values from source installations.
4. For each key, invokes the same install path as `install_skill` for `dest_target_id` (reuse shared installer logic; do not reimplement linking).
5. Classifies outcomes:
   - **installed**: newly linked
   - **skipped**: already installed, conflict, or otherwise intentionally not applied
   - **failed**: unexpected errors (include skill display label + message)
6. Persistence follows each successful install the same way `install_skill` does today (no separate all-or-nothing transaction).
7. Returns:

```ts
interface SyncTargetInstallationsResult {
  installed: number;
  skipped: number;
  failed: { storageKey: string; label: string; error: string }[];
}
```

Invalid / missing main-library skills for a source key count as **failed** with a clear reason (not silent skip), so the user can see why a source installation could not be reproduced.

### Frontend API

Expose `syncTargetInstallations(sourceTargetId, destTargetId)` in `src/api/commands.ts` (or the existing target commands module) returning the result DTO above.

### Shared Dialog — `SyncFromTargetDialog`

Props (conceptual):

- `open`, `onClose`
- `destTarget: Target`
- `candidateSources: { target: Target; installedCount: number }[]`
- `mode: 'post-create' | 'manual'`
- `onSynced: (result, nextState) => void`

Behavior:

- Title / lead copy differs by mode:
  - post-create: “已添加 {name}。” + short explanation
  - manual: “从项目内其他目录同步”
- Source `<select>` lists candidates as `{name} · {path hint} · 已装 N`, excluding `destTarget`.
- Default selection: highest `installedCount`; ties broken by target name ascending.
- Helper text: fill-missing policy; approximate new installs = source installed keys not already installed on dest (best-effort preview from current client state).
- Actions: secondary「暂时跳过」/「取消」; primary「同步安装」with loading + disable dismiss while submitting.
- On success: show summary `已安装 N，跳过 M，失败 K`; if `failed.length > 0`, list labels/errors inline; still treat batch completion as success for closing unless the entire batch failed with zero installed and all failed — in that case keep dialog open with error.
- Escape / overlay: follow existing dialog busy rules (blocked while submitting).

### Post-create follow-up

After **both** quick-add preset and custom add succeed:

1. Close `AddTargetDialog` as today (preserve one-click / simple submit).
2. Apply `onSuccess` state update so the new target exists in client state.
3. If `scope === 'project'` and there exists at least one sibling target in the same project with `installedCount > 0`, open `SyncFromTargetDialog` in `post-create` mode with `destTarget` = newly created target.
4. Otherwise do not open the dialog.

Do **not** open for global targets, when the project has no other targets, or when every sibling has zero installations.

### Target page entry

On `TargetDetail` for a project-scoped target, when at least one sibling candidate exists (`installedCount > 0`), show「从其他目录同步…」in the header action area. Opening it uses the same dialog in `manual` mode. Hide the button when there are no candidates.

### UX notes

- Creating a target never blocks on sync; sync is always a separate optional step.
- Directory page sync and post-create sync share one component and one command so behavior stays identical.

## Feature 2 — Target Detail Counts and Sort

Applies only to `TargetDetail`. Skill Hub unchanged.

### Source tree counts

For each source-tree node (including「全部」), show `installed/total` where:

- **total** = number of valid skills visible under that node for the current target list (same membership rules as today’s node filter).
- **installed** = subset of those skills whose target state is installed.

Use tabular numerals for alignment. Counts are relative to the **current target**, not global main-library install records.

Reuse / extend helpers in `sourceTreeUtils.ts` (or TargetDetail-local wrappers) rather than duplicating filter logic. Invalid skills are excluded from both numerator and denominator unless they already appear in the main valid list today — follow the same validity filter the right list uses for the primary section.

### Right-list sort

After search/source filtering of the valid-skill section:

1. Sort ascending by display name regardless of install state: `skill.name` if non-empty, else `skill.dirName` (locale-aware string compare consistent with the app’s existing locale, default user locale).
2. The separate「无效 Skill」block remains below; sort that block the same way by name/`dirName`.

## Feature 3 — Upload on Discover Tab

In `SkillHubPage`, `showUploadButton` keeps all current conditions except the `tab === 'installed'` requirement. Upload appears on both `已安装` and `可安装` when:

- at least one Hub endpoint is enabled, and
- the selected node is an enabled Hub root.

Upload dialog and backend upload path are unchanged.

## Error Handling

| Case | Behavior |
|------|----------|
| Cross-project / global sync attempt | Command error; dialog shows message |
| Source == dest | Command error |
| Partial batch failure | Summary + failed rows; no rollback |
| Post-create dialog dismissed | Destination stays with whatever was installed (usually empty) |
| No candidates | No follow-up; hide target-page sync button |
| Upload rules | Unchanged aside from tab gate removal |

## Testing

Backend:

- sync same project installs missing links and skips already installed;
- rejects different project / global / same id;
- reports failed keys for missing main-library skills;
- result counts match outcomes.

Frontend:

- post-create follow-up opens only when siblings have installations > 0;
- quick-add still closes add dialog immediately before follow-up;
- custom add uses the same follow-up path;
- shared dialog default source = max installed count;
- TargetDetail source nodes render `installed/total`;
- TargetDetail list order: installed group then uninstalled, each by name;
- Skill Hub upload visible on discover tab for enabled Hub root; still hidden for「全部」/repo/local/disabled hub.

Verification: focused Vitest + Rust tests for the new command, full suite, and production build.

## Non-Goals Recap

No file-level directory clone, no cross-project templates, no Skill Hub list re-sort in this change, no destination uninstall-to-match.
