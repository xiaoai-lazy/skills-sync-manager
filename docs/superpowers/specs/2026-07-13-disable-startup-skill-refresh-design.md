# 按来源控制启动自动刷新设计

> 日期：2026-07-13
> 状态：交互规则已确认，等待书面规格最终评审

## 目标

应用启动时立即加载并展示本地缓存，同时根据来源类型配置在后台刷新 Skill 数据。默认行为针对公司内网场景优化：GitHub 不自动刷新，GitLab 和 Skill Hub 自动刷新。

## 默认配置

在应用设置中增加启动自动刷新策略：

```text
GitHub:   false
GitLab:   true
SkillHub: true
```

旧配置缺少这些字段时直接使用上述默认值，不提升配置版本，也不执行数据迁移。配置保存在现有 `settings` 中，并通过 serde 默认值保证旧配置可读取。

每个来源类型只有一个开关，同时控制：

- 启动时刷新该类型来源的“可安装 Skill”列表。
- 启动时检查该类型来源的已安装 Skill 是否存在更新。

开关只影响启动自动刷新。用户手动点击刷新或检查更新时，仍检查全部已启用来源，包括 GitHub。

## 启动流程

```text
读取 AppState 与运行时缓存
-> 立即展示页面和缓存数据
-> 发起独立的后台启动刷新命令
-> 后端读取自动刷新策略
-> 仅刷新已开启的来源类型
-> 按来源类型合并新结果与旧缓存
-> 前端用返回结果更新展示状态
```

启动后台任务不参与首屏加载状态，不阻止窗口操作、导航或本地 Skill 浏览。应用自身的版本更新检查保持独立并继续自动执行。

## 后端接口

新增专用于启动场景的命令，例如：

```text
refresh_startup_skill_sources() -> StartupSkillRefreshResult
```

返回结果至少包含：

- 合并后的可安装 Skill 列表。
- 合并后的待更新 Skill 列表。
- 可选的来源警告，仅用于诊断，不在启动时弹出全局错误。

该命令读取配置中的三个来源开关，不接受前端临时传入的策略，避免前端状态与持久化配置不一致。

现有 `discover_skills` 和 `check_skill_updates` 保持原有“检查全部已启用来源”的语义，继续服务用户手动操作。

## 来源过滤与缓存合并

启动刷新必须按 `github`、`gitlab`、`skillhub` 三种来源类型处理，不能用过滤后的结果整体覆盖缓存。

合并规则：

- 自动刷新关闭的来源类型：完全保留旧缓存，不发起网络请求。
- 自动刷新开启且刷新成功的来源类型：用新结果替换该类型的旧缓存条目。
- 自动刷新开启但刷新失败的来源类型：保留该类型旧缓存，并记录警告。
- 一个来源类型失败不能清空其他来源类型的缓存，也不能阻止其他类型继续刷新。
- 新增、删除或停用具体来源后，继续沿用现有来源管理命令的缓存清理规则。

为避免部分失败导致缓存误删，启动刷新以来源类型为保守合并边界。某类型中任一来源刷新失败时，该类型保留旧缓存；成功结果可在下次完整成功或用户手动刷新时更新。

## 异步与并发边界

前端在 `getAppState` 完成后以不等待的方式调用启动刷新命令。后端把阻塞 HTTP、下载、解压和目录哈希放入 blocking worker，避免占用 Tauri 异步执行线程。

启动刷新沿用统一的进行中保护：

- 同一时间只允许一个启动刷新任务。
- 与手动发现或手动更新检查重叠时，后发任务返回现有的进行中错误或跳过执行。
- 启动任务失败不显示全局错误，不覆盖当前缓存。
- 用户主动操作的错误提示保持现有行为。

本次不建立通用任务系统，不增加进度条、取消机制或复杂状态机。

## 配置界面

在“来源管理”抽屉中增加“启动自动刷新”区域，提供三个开关：

- GitHub（默认关闭）
- GitLab（默认开启）
- Skill Hub（默认开启）

说明文字明确指出：这些开关只影响应用启动时的后台刷新，手动刷新始终检查所有已启用来源。

切换开关后立即持久化配置。保存失败时恢复原显示值并展示现有错误提示。修改开关不立即触发刷新，从下一次应用启动开始生效。

## 实现范围

- Rust 配置模型增加来源自动刷新设置及默认值。
- TypeScript 配置类型同步增加字段。
- 新增保存自动刷新设置的 Tauri 命令和前端 API。
- 新增启动专用的来源过滤、保守缓存合并命令。
- `useAppBootstrap` 在加载缓存后异步触发启动刷新。
- `useSkillHub` 消费合并后的发现列表和待更新列表。
- “来源管理”增加三个配置开关。

## 不在本次范围内

- 不合并或复制 `codex/update-system-refactor` 分支代码。
- 不重构应用自更新模块。
- 不建立统一更新状态机、进度系统或诊断框架。
- 不改变手动刷新接口的业务语义。
- 不给单个仓库或单个 Hub 端点增加独立自动刷新开关。
- 不增加刷新周期、延迟时间或定时任务配置。

## 测试方案

Rust 测试：

- 旧配置缺少字段时得到 GitHub 关闭、GitLab 和 Skill Hub 开启的默认值。
- 配置保存和读取能够保持三个开关。
- 来源过滤不会调用关闭类型的下载或请求钩子。
- 新结果只替换对应来源类型的缓存。
- 某类型失败时保留该类型旧缓存。
- GitLab 或 Skill Hub 失败不影响另一类型完成刷新。

前端测试：

- 启动先展示缓存，再异步调用启动刷新接口。
- 默认配置下启动刷新结果可更新 GitLab 和 Skill Hub 状态，GitHub 缓存仍保留。
- 启动刷新失败不显示全局错误，也不清空缓存。
- 三个配置开关显示正确默认值并能持久化。
- 手动刷新仍调用检查全部来源的原接口。
- 应用版本自动检查继续执行。

回归验证：

```powershell
npm test -- --run
npm run build
cargo test --manifest-path src-tauri/Cargo.toml
```

## 完成标准

- 首屏展示不等待任何 Skill 网络请求。
- 默认情况下，启动不访问 GitHub Skill 来源。
- 默认情况下，启动后台刷新已启用的 GitLab 和 Skill Hub 来源。
- 三个来源类型均可在“来源管理”中独立控制启动自动刷新。
- 自动刷新关闭或失败不会清空对应来源的旧缓存。
- 手动刷新始终检查全部已启用来源。
- 应用版本自动检查保持不变。
- 自动化测试和生产构建通过，没有新增 Rust 回归。
