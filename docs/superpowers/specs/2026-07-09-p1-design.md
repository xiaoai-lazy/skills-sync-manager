# P1 设计：CI、runtime-cache、Command 精简、App hooks

> 日期：2026-07-09  
> 状态：已批准（brainstorming）  
> 关联：[架构审查](./2026-07-08-architecture-review.md)、[P0 设计](./2026-07-08-p0-design.md)  
> 方案选择：**风险递增四阶段**（CI → runtime-cache → Command → App hooks）

---

## 1. 范围与目标

### 1.1 本轮做什么（仅 P1）

| 编号 | 目标 |
|------|------|
| **P1-4** | PR / push 到主开发分支时跑 `npm test` + `cargo test` |
| **P1-3** | `skillDiscoverCache` / `skillUpdateCache` 迁出 `config.json`，写入独立 `runtime-cache.json` |
| **P1-2** | 渐进精简 mutation 返回的 `AppState`；减少每次 mutation 全量 `list_skills` |
| **P1-1** | 用自定义 hooks 拆分 `App.tsx`，去掉 hub ↔ app 双向同步，不引入状态库 |

### 1.2 交付节奏

一份设计 + 一份实施计划；落地为 **4 个可独立合并的阶段/PR**，顺序固定：

1. CI（P1-4）  
2. runtime-cache（P1-3）  
3. Command 精简（P1-2）  
4. App hooks（P1-1）

### 1.3 技术选型（已确认）

| 议题 | 选择 |
|------|------|
| 前端状态 | 自定义 hooks / 模块，不引入 Zustand 等库 |
| Command 返回 | 渐进式精简 `AppState` + `mergeAppState`；发现/扫描保持专用结果类型 |
| 运行时缓存 | 独立 `app_data/runtime-cache.json` |
| CI | 单 job：`npm test` + `cargo test` |

### 1.4 明确不做（整轮 P1）

- 全部 **P2**（类型 codegen、CSS 拆分、并发重构、API 冗余清理、SourceTree 解耦、迁移 ADR 等）
- 引入 Zustand / Redux 等状态库
- mutation 改为「只返回 void + 前端全量重拉」
- 发现列表改为纯内存、不落盘
- 改产品 UI 视觉 / 新功能
- `tauri build` / lint 门禁（除非已有零成本脚本）

### 1.5 成功标准（整轮结束时）

1. CI 在 PR 上拦住前后端测试失败  
2. `config.json` 不再因 discover/update 列表膨胀；升级后自动迁移缓存  
3. 常见 mutation 的 IPC 载荷与主库扫描次数明显下降（可测）  
4. `App.tsx` 降为组装层；hub / target / dialog 状态各有单一写入路径

## 2. 阶段 1 — CI（P1-4）

### 2.1 新增文件

`.github/workflows/ci.yml`

### 2.2 触发

- `pull_request` 与 `push`，分支：`main`（或仓库默认主分支）以及 `feat/v0.6`（当前开发线；该分支合并关闭后从 workflow 中移除）
- **不**触发：`v*` tag（仍只走 `release.yml`）

### 2.3 Job

单 job，runner：`windows-latest`（与本地及 junction 相关测试一致）。

```
checkout
→ setup Node (lts) + npm ci
→ npm test
→ setup Rust stable
→ cargo test --manifest-path src-tauri/Cargo.toml
```

本阶段不做多 OS 矩阵；若后续过慢再拆。

### 2.4 明确不做

- 不跑 `tauri build` / 打包  
- 不新增 lint/format 门禁  
- 不改 `release.yml`  

### 2.5 验收

- 新开 PR 出现 CI check；故意弄坏测试时 CI 失败  
- 与 release 工作流互不干扰

## 3. 阶段 2 — runtime-cache（P1-3）

### 3.1 落盘结构

```
app_data/
├── config.json              ← 用户设置、targets、installations、skill_records、repos、hub endpoints…
├── runtime-cache.json       ← 【新增】发现/更新缓存
└── repo-cache/              ← P0 已有，本阶段不动
```

`runtime-cache.json` 形状：

```json
{
  "version": 1,
  "skillDiscoverCache": { "skills": [], "fetchedAt": "..." },
  "skillUpdateCache": { "updates": [], "checkedAt": "..." }
}
```

字段形状与现有 `SkillDiscoverCache` / `SkillUpdateCache` 一致，只换存储位置。

### 3.2 读写职责

| 操作 | 行为 |
|------|------|
| `config_store::load` | 读 `config.json`；若仍含旧缓存字段 → 迁入 `runtime-cache.json` → 清空 config 内字段并 save config |
| discover / checkUpdates | 更新内存后 **只写** `runtime-cache.json`，不因此整份 save config（除非同时改了用户配置） |
| `get_app_state` / 启动 | 合并 config + runtime-cache → 组装 `AppState`（兼容层可把缓存填回现有字段） |
| 文件缺失 / 损坏 | 视为空缓存；discover 失败时不覆盖成空文件（保留上次成功写入） |

### 3.3 迁移

- load 时若 `config` 内 discover/update 缓存非空：写入 runtime-cache；若 runtime 已有内容，**以 config 侧旧字段为准覆盖一次**，再清空 config 字段并持久化 config  
- `config.json` 继续 tmp + backup；`runtime-cache.json` 用 tmp + rename 原子写  

### 3.4 API / 前端

- 本阶段尽量不改前端消费路径：后端**仅在内存中**组装 `AppState` 时，可从 runtime-cache 填入 `skillDiscoverCache` / `skillUpdateCache` 供 UI 读取  
- **禁止**在 save `config.json` 时再把这两项写回磁盘；持久化只走 `runtime-cache.json`  
- 阶段 3/4 再考虑类型上摘掉或挪到 `AppState.runtimeCache`  

### 3.5 明确不做

- 不拆成两个缓存文件  
- 不做 LRU / 体积上限  
- 不改 `repo-cache`  
- 不做 Hub REST 文件缓存  

### 3.6 验收

- 升级后 `config.json` 不再含大段 skills 列表；`runtime-cache.json` 存在且正确  
- 二次启动仍能展示上次 discover / updates  
- Rust 单测：迁移、原子写、损坏兜底  

### 3.7 文档

- README 补一句：`runtime-cache.json` 用途与可手动删除（本阶段实施时改）

## 4. 阶段 3 — Command 精简（P1-2）

### 4.1 策略

**`build_app_state` 分层：**

| 模式 | 行为 | 用于 |
|------|------|------|
| `Full` | 扫主库 + 算 selected target states | `get_app_state`、启动、主库文件树变更 |
| `Light` | 不重新 `list_skills`；返回更新后的 config + selectedTargetId +（若有）当前目标 `selectedTargetSkills`；`skills` 可省略/null | 纯链接安装卸载、目标/项目元数据 mutation |

**主库是否变更：**

- **未改主库文件树**（install/uninstall 到目标、改 target/project 元数据）→ Light；前端 `mergeAppState` 保留 `prev.skills`  
- **改了主库**（`delete_main_skill`、`install_hub_skill`、`set_main_skills_dir` 等）→ Full 或至少重扫 skills  

**前端：**

```
next = await installSkill(...)
setAppState(prev => mergeAppState(prev, next))
```

`mergeAppState`：`next.skills` 为 null/undefined 时保留 `prev.skills`；其余以 next 为准。阶段 3 可先放在 `App.tsx` 旁工具模块，阶段 4 hooks 复用。

类型上：Light 路径的 IPC 返回可用 `AppState` 的部分字段（例如 `skills?: SkillView[] | null`），或单独 `AppStatePatch`；前端 merge 后仍得到完整 `AppState`。实施计划选定一种并保持前后端一致。

**已专用返回类型**（`discover_skills` → `DiscoverSkillsResult` 等）保持不变。

**与 runtime-cache：** Full/Light 组装时从 runtime-cache 填 discover/updates；类型上是否挪出 `config` 可选，改动面小则做，否则留给阶段 4。

### 4.2 明确不做

- 不引入文件系统 watch / 按 skill 订阅  
- 不把所有 mutation 改成 `void`  
- 不做长期进程内 TTL 扫描缓存（若同请求链仍重复扫，仅允许**请求内**短缓存）  

### 4.3 验收

- 安装/卸载到目标：不触发主库全量 `list_skills`（测试可证）  
- 删除主库 skill / 改主库路径：skills 列表一致  
- `mergeAppState` 单测 + 现有前端测试全绿

## 5. 阶段 4 — App hooks 拆分（P1-1）

### 5.1 目标结构

```
src/
├── App.tsx                      ← 布局 + hooks 组装（尽量薄）
├── hooks/
│   ├── useAppBootstrap.ts       ← getAppState、loading/error、migration toast、启动后台任务
│   ├── useAppState.ts           ← appState + mergeAppState + 选中目标/项目/主视图
│   ├── useSkillHub.ts           ← discover/updates 列表；从 appState 单向派生
│   ├── useTargetActions.ts      ← install/uninstall/toggle、目标 CRUD、pendingSkillKey
│   ├── useProjectActions.ts     ← 项目 CRUD / 展开态（可与 target 合并若过碎）
│   └── useAppDialogs.ts         ← 各类 dialog 开关与草稿
└── utils/mergeAppState.ts       ← 复用阶段 3
```

文件名实施时可微调；原则：按职责拆分，不引入状态库。

### 5.2 状态归属

| 状态 | 归属 |
|------|------|
| `appState`（经 merge） | `useAppState` |
| `selectedTargetId` / `selectedProjectId` / `expandedProjectIds` / `mainView` | `useAppState`（或小型 navigation hook） |
| `discoverSkillsList` / `pendingUpdates` / hub 扫描元数据 | `useSkillHub` |
| `pendingSkillKey`、删除用 storageKey | `useTargetActions` |
| 各类 dialog open/data | `useAppDialogs` |
| `loading` / `error` / migration toast | `useAppBootstrap` |

### 5.3 单向数据流

```
getAppState / mutation(merge)
        │
        ▼
    appState   ← skillRecords 等唯一事实来源
        │
        ├── useSkillHub：启动时从 cache 灌列表；后台刷新只更新列表；
        │                写回 records 只经 useAppState patch
        └── 页面组件：读 appState + hub 列表
```

规则：

1. `skillRecords` 只存在于 `appState.config`（或等价单一处）  
2. 删除 `applyHubState` ↔ `buildHubStateFromAppState` 对 records 的双向拷贝；`hubState` 改为派生视图或仅保留扫描元数据  
3. 不引入全局 Context/Provider（本阶段）；hooks 返回值在 `App` 组装  

### 5.4 测试

- `app.test.tsx` 保持黑盒 render `<App />`  
- `mergeAppState`、hub 派生纯函数补单测  
- 不强制每个 hook 上 RTL  

### 5.5 明确不做

- 不引入 Zustand/Redux/全局 store  
- 不重做 UI、不拆 `SkillHubPage` 内部（P2）  
- 不做 CSS 模块化  

### 5.6 验收

- 现有 Vitest 关键路径全绿  
- 代码中不再存在 hub ↔ app 对 `skillRecords` 的双向同步对  
- `App.tsx` 以 hooks 组装为主，业务逻辑在 `hooks/`

## 6. 跨阶段依赖、错误处理与测试

### 6.1 依赖顺序

```
阶段1 CI          ──独立，可先合
阶段2 runtime-cache ──► 阶段3 组装 AppState 时读 runtime-cache
阶段3 Command/merge ──► 阶段4 复用 mergeAppState，消费 Light 返回
阶段4 App hooks     ──依赖阶段 2+3 数据形状稳定
```

阶段 4 不要在阶段 2 未合时强行改 cache 字段归属。

### 6.2 错误处理

| 场景 | 行为 |
|------|------|
| CI 失败 | 阻断 PR；不改 release |
| `runtime-cache.json` 缺失/损坏 | 空缓存；不阻断 config load |
| runtime-cache 写入失败 | 保留旧文件；内存结果可返回 + warning |
| 迁移时 config 与 runtime 都有缓存 | **config 旧字段覆盖写 runtime 一次**，再清空 config 字段 |
| Light 返回时 `prev` 为空 | 整包采用 `next`，或触发一次 Full `get_app_state` |
| 后台 discover / checkUpdates 其一失败 | 互不清空对方列表（P2-3 不做） |

### 6.3 测试策略

| 阶段 | 重点 |
|------|------|
| 1 | `ci.yml`；命令与本地一致 |
| 2 | 迁移、原子写、损坏兜底、discover 只写 runtime-cache |
| 3 | install/uninstall 不全量扫主库；主库变更走 Full；`mergeAppState` |
| 4 | `app.test.tsx` 黑盒；去掉双向同步后的派生函数 |

整轮验证：

```bash
npm test
cargo test --manifest-path src-tauri/Cargo.toml
```

### 6.4 风险与缓解

| 风险 | 缓解 |
|------|------|
| Windows CI 慢/偶发 | 先单平台；必要时再加 timeout，本轮不上矩阵 |
| Light 漏字段导致 UI 旧数据 | `mergeAppState` 明确字段规则；主库变更强制 Full |
| 阶段 4 大面积测试红 | 先抽纯函数与 merge，再搬 handler；每步跑 `app.test.tsx` |
| 旧配置无缓存字段 | 缺省为空；runtime 缺失则空列表 + 后台刷新 |
